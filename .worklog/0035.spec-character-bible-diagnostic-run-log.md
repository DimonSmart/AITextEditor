## Goal

Persist a compact detailed diagnostic log for each Character Bible workflow run while keeping UI progress short and user-facing.

## Context

The operation UI already shows the broad workflow order, candidate counts, retries, and review verdicts. That is useful for live feedback, but it is not enough to reconstruct identity resolution and patch diagnostics after a run. The missing details include vector-search queries, search hits and scores, resolver reasons, patch proposal/review diagnostics, retry points, malformed responses, and technical protocol warnings.

Application settings already live under `%LOCALAPPDATA%\AITextEditor`, so diagnostic logs should live beside them under `%LOCALAPPDATA%\AITextEditor\logs`.

## Scope

- Add application file logging to `%LOCALAPPDATA%\AITextEditor\logs\aitexteditor.log`.
- Add a per-run Character Bible log at `%LOCALAPPDATA%\AITextEditor\logs\character-bible\character-bible-run-YYYYMMDD-HHMMSS.log`.
- Use short log lines: `HH:mm:ss LEVEL event message/key-values`.
- Log workflow, collect, extraction, resolution, vector search, patch proposal, patch review, and patch apply events with stable event names.
- Include vector-search query, hit count, hit names, entry ids, and scores in the run log.
- Include resolver decisions, selected entry ids, and resolver reasons in the run log.
- Include retry, malformed response, contract validation, and protocol warning/error diagnostics in the run log.
- Show only one extra UI progress row with the detailed log path and copy action.

## Non-goals

- Do not show per-candidate search hits in UI progress.
- Do not log embedding vectors, API keys, authorization headers, full prompts, full response bodies, full book text, or full dossier snapshots by default.
- Do not add Serilog, NLog, or another external logging package only for this feature.
- Do not add semantic identity heuristics in C#; the LLM remains responsible for identity similarity decisions.
- Do not change public workflow APIs unless the existing internal wiring cannot safely carry the run logger.

## Plan

1. Add log path helpers and a simple file logger provider for the global app log.
2. Add a Character Bible run log context/logger with compact value formatting helpers.
3. Create the run logger at workflow start and report its path once through UI progress.
4. Thread the run logger through workflow executors, extractor, resolver/search adapter/vector search, and patcher.
5. Add focused tests for log paths, line formatting, and representative Character Bible diagnostic events.

## Pause points

- Stop if the project already has another file logger that conflicts with the new provider.
- Stop if `%LOCALAPPDATA%` is unavailable for the application runtime.
- Stop if passing the run logger through the workflow requires a broad public API rewrite.
- Stop if raw LLM body logging would risk writing secrets outside the existing redaction/truncation path.

## Acceptance criteria

- UI progress remains short and includes a copyable detailed log path at the start of a run.
- A per-run Character Bible log file is created with a sortable `YYYYMMDD-HHMMSS` run id.
- The run log shows each workflow stage in order.
- Each resolved candidate has candidate id/name/pointers, search query, search hits, and resolver decision entries.
- Patch proposal, retry, review, and apply/skip diagnostics are visible in the run log.
- Log lines use short timestamps and levels, and do not include thread id, user, machine, or process id.

## Verification

Commands:

```bash
dotnet build
dotnet test --filter "Category!=Manual"
```

Manual check:

```text
Run Character Bible workflow on a small fragment and inspect the generated character-bible-run-*.log for resolve.search, resolve.search.hits, resolve.decision, patch.proposal.retry, patch.review.result, and patch.apply/skipped entries.
```

## Outcome

Implementation notes:

- The per-run logger is carried through an async workflow scope so internal agents, vector search, and patching can write to the same run log without widening public workflow APIs.
- The existing simple file logger provider was updated to the compact `HH:mm:ss LEVEL message` format and is wired in Web startup for `%LOCALAPPDATA%\AITextEditor\logs\aitexteditor.log`.
- Character Bible UI progress gets one always-visible copyable detailed-log-path event; search hits and raw trace details stay in the file log.

Verification:

```bash
dotnet build
dotnet test --filter "Category!=Manual"
```

Both commands passed. The first build attempt found a running `AiTextEditor.Web` process locking Web output DLLs; that process was stopped and the build was rerun successfully.
