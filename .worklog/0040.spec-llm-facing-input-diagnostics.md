## Goal

Make Character Bible run logs show the dynamic input model visible to each LLM call without logging full system prompts or static instruction wrappers.

## Context

Compact event summaries are not enough to diagnose whether the model received the expected paragraph text, evidence, current profile, and proposed additions. Logging full prompts would add constant noise and duplicate static instructions. The useful diagnostic boundary is the projected input DTO serialized for the model.

## Scope

- Add untruncated block logging for dynamic LLM-facing input DTOs.
- Log input DTOs for extract, resolve, patch proposal, and patch review calls.
- Keep code correlation ids in log headers and outside compact patch/review DTOs.
- Log resolver search tool calls and returned visible hit metadata separately from correlation ids.
- Ensure reviewer input contains only evidence referenced by proposal additions.
- Add patch proposal contract diagnostics for top-level keys, forbidden keys, evidence count, and empty evidence texts.
- Add focused tests that assert concrete paragraph text appears in logged input blocks.

## Non-goals

- Do not log full system prompts.
- Do not log static prompt instructions or retry boilerplate as LLM input.
- Do not log full dossiers or internal candidate objects.
- Do not add alias handling to patch proposal input.
- Do not truncate LLM input model blocks through `ShortText`.

## Plan

1. Add an atomic debug block method to the Character Bible run logger and a shared LLM input serializer.
2. Project extract, resolve, patch proposal, and patch review calls through explicit input DTOs used by both serialization and logging.
3. Separate resolver search tool interaction logs from correlation metadata.
4. Add patch proposal contract diagnostics and focused tests.

## Pause points

- Stop if logging dynamic input models would require logging secrets or authorization data.
- Stop if compact DTO changes require a broad public API migration.

## Acceptance criteria

- Run logs contain untruncated `extract.llm.input`, `resolve.llm.input`, `patch.proposal.llm.input`, and `patch.review.llm.input` blocks.
- Logged blocks contain the actual dynamic DTO serialized for the corresponding model call.
- Full system prompts and static prompt instructions are absent from the new log events.
- Extract logs contain full paragraph texts.
- Resolve logs contain candidate name, gender, aliases, and evidence texts without internal candidate ids inside the model.
- Patch proposal logs contain only target, current profile, and new evidence inside the model.
- Patch review logs contain only referenced evidence.
- Resolver tool search and tool result events distinguish visible hit ids from code correlation ids.
- Tests assert concrete paragraph text inside logged model blocks.

## Outcome

Implementation notes:

- Extract and resolve logging use the same explicit DTO instances passed to JSON serialization.
- To keep those DTOs strictly LLM-facing, extract input no longer carries the static `task` marker and resolve input no longer carries internal `candidateId`; resolve evidence is nested under `candidate.evidence`.
- Block logging preserves explicit `null` profile fields and does not use `ShortText`.
