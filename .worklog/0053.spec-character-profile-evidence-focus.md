# 0053 SPEC Character profile evidence focus

Type: spec

- none

## Goal

Build compact `profile.update` evidence focus around the target character mention instead of around a generic paragraph anchor.

## Context

The compact `newEvidence` contract removed repeated full paragraph text, but the focused text can now be worse than the old full paragraph when several characters share one paragraph. If the focus is built around the first paragraph anchor, the profile update agent may not see the target character's observed form or may attribute nearby facts to the wrong character.

## Scope

- Select `focusedText` for each profile update evidence item using observed name forms from the current resolved candidate.
- Prefer exact observed-form matches inside the paragraph text.
- Try case-insensitive and `ё`/`е` normalized matching before falling back.
- Keep compact `newEvidence`; do not restore full paragraph or label-style repeated evidence text.
- Log normal focus diagnostics through `profile.evidence.focus`.
- Log fallback diagnostics through `profile.evidence.focus.fallback`.
- Add a profile-update prompt rule that facts must be clearly attributed to the target character.
- Add tests for multi-character paragraphs and fallback diagnostics.

## Non-goals

- Do not change extraction overlap or pointer traversal.
- Do not change identity resolution.
- Do not add full paragraph text back to `newEvidence`.
- Do not infer character-specific actions with heuristics beyond choosing a local mention window.

## Plan

1. Carry candidate/dossier identity into profile evidence focus building.
2. Find the best observed-form mention in the source paragraph and build a bounded focus window around it.
3. Add focus and fallback log events.
4. Update prompt guidance and focused tests.

## Pause points

- Stop if observed-form matching would require changing persisted Character Bible storage.

## Acceptance criteria

- For ordinary profile evidence, `focusedText` contains the target character's observed form when that form is present in the paragraph.
- Multi-character paragraphs can produce different focused text for different target characters.
- Fallback focus is logged when no observed form can be found.
- Compact `newEvidence` remains `pointer`, `focusedText`, optional `contextBefore`, and optional `contextAfter`.
- The prompt warns the model not to transfer facts from nearby characters.

## Outcome

Implementation notes:

- Focus selection first tries observed forms whose extraction evidence points to the current pointer, then the remaining observed forms for the candidate.
- The focused window starts at the sentence containing the target mention and may include the following sentence when it stays within the compact limit. It intentionally does not include the previous sentence, because that reproduced the regression where focus began with another character.
- Fallback uses the previous compact focus behavior and writes `profile.evidence.focus.fallback`; every evidence item also writes `profile.evidence.focus`.
