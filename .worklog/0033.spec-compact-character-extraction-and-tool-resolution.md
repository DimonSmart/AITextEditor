## Goal

Simplify the Character Bible extraction LLM contract to compact local character candidates and replace the old identity resolver path with one-candidate-at-a-time tool-based identity resolution.

## Context

The current extraction contract still carries nested character, alias, and evidence objects that make the LLM copy text and resemble downstream API models. Identity resolution also keeps older deterministic/fallback behavior around the agent path. The intended split is simpler: the extraction LLM performs local semantic grouping, C# performs mechanical validation and mapping, and the identity resolver LLM resolves one candidate against archive search hits through a string search tool.

## Scope

- Replace the LLM-facing extraction response with `characters[]` containing `name`, `gender`, `aliases`, and `pointers`.
- Update the extraction prompt so it does not request excerpts, profile text, alias evidence, character evidence, existing/new decisions, or candidate ids.
- Adapt compact extraction results to the existing internal candidate model by validating fields, checking pointers, generating candidate ids, building evidence from paragraph text, and building alias examples from aliases.
- Keep only exact repeated model item suppression in C#; do not add semantic character dedupe.
- Replace the active identity resolver with a tool-based agent that handles one candidate per call.
- Add a string-query character archive search tool adapter returning compact archive hits.
- Map `existing`, `new`, `ambiguous`, `identity_conflict`, and `defer` decisions into the existing commit plan flow.
- Remove the old resolver/fallback path instead of keeping a feature flag or compatibility mode.
- Update focused tests for extraction mapping, pointer validation, repeat suppression, resolver input/output, search adapter, and lack of batch resolver calls.

## Non-goals

- Do not rewrite the workflow engine.
- Do not change patching or commit stages unless the new resolver contract forces a narrow integration change.
- Do not add automatic card splitting.
- Do not implement semantic identity merge in C#.
- Do not keep the old resolver as fallback.
- Do not support old and new extraction schemas in parallel beyond local migration.
- Do not add a `GetParagraphs` tool for identity resolution.

## Plan

1. Inspect current extraction DTOs, post-processing, resolver, search service, and tests.
2. Replace compact extraction DTOs and prompt resource.
3. Update mapper/postprocessor validation and internal candidate mapping.
4. Add the tool-based identity resolver client, agent, response contracts, and search tool adapter.
5. Route `CharacterBibleResolver` through the new one-candidate resolver path and remove unused old resolver code.
6. Update tests and run repository checks.

## Pause points

- Stop if the internal candidate model requires an excerpt and paragraph text cannot be used as `anchorExcerpt`.
- Stop if `CharacterArchiveSearchService` cannot be adapted to string-query compact hits without changing its core semantics.
- Stop if removing the old resolver requires rewriting patching or commit stages.
- Stop if compact extraction requires changing the patching prompt.
- Stop if JSON schema generation cannot handle the resolver enum and requires string-based validation.

## Acceptance criteria

- Extraction LLM responses contain only `characters/name/gender/aliases/pointers`.
- Empty names and invalid pointers are rejected with diagnostics.
- `unknown` gender is accepted.
- Downstream pipeline still receives internal candidate models in its expected format.
- Resolver processes exactly one candidate per agent call.
- Resolver can call a string-query search tool with `query` and `limit`.
- Search tool returns compact hits with entry id, name, gender, aliases, identity, and score.
- Existing and ambiguous resolver decisions validate referenced entry ids.
- C# does not perform semantic identity merge.
- The old resolver/fallback path is removed or no longer usable.

## Outcome

Implemented the compact extraction contract and replaced the active identity resolution path with one-candidate tool-based resolution.

Notable implementation details:

- extraction model output is now `characters[]` with `name`, `gender`, `aliases`, and `pointers`;
- post-processing validates pointers against the input batch, rejects invalid local candidates with progress diagnostics, builds evidence from paragraph text, and only suppresses exact repeated model items;
- old `canonicalName/alias.evidence/character.evidence/excerpt` extraction DTOs were removed from the LLM-facing contract;
- identity resolution now receives one candidate per call and registers `search_characters(query, limit)` as a model tool;
- the previous suspect archive resolver fallback and deterministic resolver path were removed from the active code path;
- tests use an explicit search-backed fake identity resolver instead of relying on deterministic fallback behavior.
