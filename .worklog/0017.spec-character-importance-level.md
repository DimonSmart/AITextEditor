## Goal

Add a stable nullable character importance level to Character Bible dossiers and derive it from transient resolved-character activity during generation.

## Context

The Character Bible workflow already collects text fragments, extracts candidates, resolves candidates, and commits dossiers. The UI stores structured JSON companion files and shows character cards. The new importance marker should help users see whether a character is episodic, minor, supporting, or main without storing raw mention evidence or making later incremental refreshes rewrite stable classifications.

## Scope

- Add `int? ImportanceLevel` to the persistent `CharacterDossier` model.
- Normalize persisted importance values so missing values stay `null`, invalid low values become `null`, and values above `10` clamp to `10`.
- Add a domain helper that maps levels to labels and converts transient activity scores to levels `1..10`.
- Count transient activity only for successfully resolved `New` and `Existing` resolver decisions during a workflow run.
- Apply full-generation policy: set importance only when the dossier has `null`.
- Apply incremental policy: never change existing dossier levels, and cap new-character levels at `4`.
- Show the label in the Character Bible UI and support nullable manual editing if it fits the existing edit dialog shape.
- Add focused tests for legacy load, generation policies, ambiguous decisions, incremental cap, and label mapping.

## Non-goals

- Do not persist mention counts, raw mentions, paragraph pointers, evidence, activity scores, confidence, or history.
- Do not ask the LLM to count mentions.
- Do not change extraction prompts or response models for mention counting.
- Do not add alias-based full-text recalculation in this change.
- Do not recalculate all existing character importance levels during partial refreshes.
- Do not add weighted scoring, decay, confidence, or complex statistics.

## Plan

1. Find the current Character Bible model, workflow, commit, service, UI, and tests.
2. Extend `CharacterDossier` and persistence normalization with `ImportanceLevel`.
3. Add a small helper for label mapping and score-to-level conversion.
4. Thread a transient activity accumulator through resolver or commit logic without serializing it.
5. Update full and incremental generation behavior according to the requested policy.
6. Update UI display and edit behavior if the existing dialog allows a small localized change.
7. Add focused unit tests and run repository checks.

## Pause points

- Stop if the project already has another durable character importance, priority, or rank concept.
- Stop if full generation cannot be reliably distinguished from incremental generation.
- Stop if resolver decisions do not expose enough information to distinguish `New`, `Existing`, and `Ambiguous`.
- Stop if adding `ImportanceLevel` breaks existing JSON or YAML serialization.
- Stop if UI editing requires a broad form rewrite.

## Acceptance criteria

- `CharacterDossier` contains nullable `ImportanceLevel`.
- Old companion files without `importanceLevel` load with `ImportanceLevel == null`.
- Generation uses only transient activity scores.
- Activity scores are not persisted.
- Full generation sets a level only when the dossier level is `null`.
- Incremental update does not change existing character levels.
- Incremental update caps new character levels at `4`.
- The UI displays the textual label.
- Tests cover the main rules.
- `dotnet test` passes.

## Outcome

Implementation kept activity scoring transient inside the generator/workflow commit path. The committed dossier schema stores only `ImportanceLevel`; workflow output does not expose activity scores.

The existing Character Bible edit dialog accepted a small nullable numeric field, so manual editing was included in this change.

Verification note: plain `dotnet test AiTextEditor.sln` starts manual LLM functional tests and timed out twice. The timed-out runs left `dotnet test`/`testhost` processes, which were stopped before rerunning checks. The non-manual solution check passed with:

```bash
dotnet test AiTextEditor.sln --no-build --filter "Category!=Manual"
```
