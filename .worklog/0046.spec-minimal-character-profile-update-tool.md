## Goal

Make the Character Bible profile-update tool contract contain only the data that changes the persisted character profile.

## Context

The profile update agent already receives a compact input with `target`, `currentProfile`, and `newEvidence`. The remaining noise is in the tool-call payload: `reason`, `status`, and field-level `evidencePointers` are generated by the model but are not persisted in the final dossier and are not needed to apply a profile-field replacement.

The profile-update stage should be determined by tool calls:

- no tool calls means no profile change;
- a valid tool call means the field replacement should be applied.

## Scope

- Replace the profile update tool args with a minimal `field` and `value` contract.
- Remove `reason`, `status`, and `evidencePointers` from the LLM-facing profile update tool schema.
- Update the profile-update prompt so the model calls the tool only for meaningful additions or corrections and gives a complete replacement value.
- Keep applying complete replacements for `appearance`, `statusAndCompetence`, `psychologicalProfile`, and `speechAndCommunication`.
- Treat zero valid tool calls as no change.
- Validate supported fields and non-empty values.
- Skip same-value replacements after trimming so they are not counted as meaningful updates.
- Update run logs to include character id, name, field, value length, applied count, and no-change events without model-generated reasons, status, or evidence pointers.
- Keep the final `character-bible.json` schema unchanged.

## Non-goals

- Do not change character grouping strategy.
- Do not change parallel profile update execution.
- Do not change group size.
- Do not change overlap paragraph behavior.
- Do not change the shape of `CharacterProfileUpdatePromptInput` unless the implementation requires it.
- Do not change which characters are selected for profile update.
- Do not add field-level citation storage.
- Do not add debug reasons by default.

## Plan

1. Update the tool adapter contract, validation, and logs.
2. Update the profile update model client and prompt.
3. Update workflow logging for applied and no-change results.
4. Update focused unit tests and run repository checks with manual local-LLM tests excluded.

## Pause points

- Stop if removing field-level evidence pointers would require changing persisted dossier storage.
- Stop if the Agent Framework cannot expose a useful tool schema for a two-field args contract.

## Acceptance criteria

- `profile.update.tool.call` logs do not contain `reason`, `status`, or `evidencePointers`.
- The profile update tool schema contains only `field` and `value`.
- If the model has no meaningful profile update, it calls no tools.
- The final character bible JSON schema is unchanged.
- Profile fields continue to update through complete replacement values.
- For the small Neznaika profile test run, Znaika still receives `statusAndCompetence` and `psychologicalProfile`.
- Characters without new profile data produce `profile.update.no_change` without a profile update tool call.
- Logs retain enough diagnostic data: character id, name, field, value length, applied count, and no-change count.
