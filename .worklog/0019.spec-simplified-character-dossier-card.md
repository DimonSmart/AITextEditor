## Goal

Simplify the Character Bible dossier card so generated and stored character profiles contain only compact, durable character attributes and do not store generic summaries or weak relationship facts.

## Context

The structured profile added in 0018 still keeps `description` on the dossier and `keyRoleBonds` inside the profile. In generated output these fields duplicate other profile sections or turn into noisy lists of plot facts, name variants, and weakly normalized relationships. The project has no saved character bible files that require the old JSON schema, so this change can update the schema directly instead of adding migration support.

`backgroundStatusEducation` has useful content, but the name is too narrow because the field stores social status, profession, role, competencies, and group position, not only education.

## Scope

- Remove `description` from the persistent character dossier model, extraction DTOs, prompts, UI, Markdown rendering, tests, samples, and generated JSON.
- Remove `profile.keyRoleBonds` and the role bond model when no longer used.
- Rename `profile.backgroundStatusEducation` to `profile.statusAndCompetence` everywhere.
- Keep the remaining profile fields: `appearance`, `statusAndCompetence`, `psychologicalProfile`, and `speechAndCommunication`.
- Update typed response validation and prompts so new character extraction writes only the new profile shape.
- Update the Character Bible card, edit dialog, search, completeness, and sample JSON to match the simplified schema.

## Non-goals

- Do not add `storyFunction` or any replacement relationship, role, or plot-function field.
- Do not solve entity merge or name normalization.
- Do not change the importance-level algorithm.
- Do not change `aliases` or `aliasExamples`.
- Do not rewrite the whole Character Bible workflow.
- Do not add legacy JSON migration or compatibility UI for the old profile schema.

## Plan

1. Find all references to the old dossier/profile fields across code, prompts, schemas, UI, tests, samples, and cassettes.
2. Update core models and normalization/merge/search/rendering behavior to the four-field profile.
3. Update extraction contracts, prompt text, and validation.
4. Update Character Bible UI card and edit dialog.
5. Update tests, fixtures, and sample JSON.
6. Run repository checks.

## Pause points

- Stop if `description` is used as a separate public API concept outside Character Bible dossiers.
- Stop if the typed response schema is generated from an external contract that cannot be changed locally.
- Stop if removing `keyRoleBonds` breaks a separate scenario explicitly built around character relationships.

## Acceptance criteria

- Generated character JSON has no `description`.
- Generated character JSON has no `profile.keyRoleBonds`.
- Generated character JSON has no `profile.backgroundStatusEducation`.
- Generated character JSON includes `profile.statusAndCompetence`.
- The UI does not show the generic description or key role bonds section.
- The UI labels the renamed field as `Статус и компетенции`.
- No `storyFunction` or equivalent replacement field is added.
- Removed role-bond types do not remain as dead code.
- `dotnet build AiTextEditor.sln` passes.
- `dotnet test --filter "Category!=Manual"` passes.

## Verification

Commands or manual checks:


## Outcome

Implementation removed the old fields from the persisted dossier model, typed extraction response, generation prompt, merge path, UI card/editor, Markdown projection, search, tests, and sample JSON. `CharacterProfile` is now the four-field shape: `appearance`, `statusAndCompetence`, `psychologicalProfile`, and `speechAndCommunication`.

The old cassette JSON fixtures were deleted instead of migrated because their recorded model responses encode the removed extraction contract. Non-manual tests do not depend on them after this change.
