# 0037 SPEC LLM contract rules

Type: spec

Replaces:
- none

## Goal

Define the project rules for typed input and output contracts between workflow code and LLM agents.

The main rule: LLM contracts are not REST DTOs. They contain only the fields the model needs for the current decision.

## Context

The Character Bible workflow uses typed models and Microsoft Agent Framework / Microsoft.Extensions.AI structured outputs. Some workflow models contain fields useful for C# code, logging, diagnostics, persistence, or UI, but not useful for the model's reasoning.

If those fields are sent to the LLM, they add noise and can make the model reason from technical metadata, temporary identifiers, debug values, or full internal objects. The contract should be a deliberate projection, not a trimmed copy of an internal model.

The project should rely on typed C# contracts for structured outputs when the Agent Framework path supports them. These rules define the semantic contract shape and do not require hand-written JSON Schema for nullable fields that can be derived from C# types.

## Applicability

These rules apply when adding or changing:

- LLM input contracts;
- LLM structured output contracts;
- tool call input and output contracts exposed to an LLM;
- projection models that translate internal workflow state into LLM-facing data.

They do not apply to persistence models, UI view models, diagnostic log models, or public REST-style contracts unless those models are also sent directly to an LLM.

## Contract rules

### Separate internal models from LLM contracts

Workflow models may contain IDs, statuses, audit data, debug values, statistics, runtime metadata, persistence metadata, and UI state.

LLM contracts must contain only the data the model needs for one concrete task. They are separate projection models, not shortened internal DTOs.

### Keep input minimal

For each agent, ask:

```text
Which fields does the model need to make this specific decision?
```

Do not send fields that do not affect the decision, including temporary candidate IDs, batch indexes, durations, retry counts, diagnostic metadata, UI state, persistence metadata, full workflow objects, or fields added "for later".

### Keep nesting meaningful

Prefer flat contracts unless nesting represents a real semantic group such as `candidate`, `dossier`, `evidenceContexts`, or `searchHits`.

Bad:

```json
{
  "request": {
    "workflow": {
      "run": {
        "candidate": {
          "identity": {
            "name": "Ponchik"
          }
        }
      }
    }
  }
}
```

Good:

```json
{
  "candidate": {
    "name": "Ponchik",
    "aliases": ["Ponchika"],
    "pointers": ["1.1.1.p4"]
  }
}
```

### Do not design LLM contracts like REST DTOs

REST DTOs often carry extra fields for clients, compatibility, future extension, or convenience. That is harmful in an LLM prompt when the field is not part of the reasoning task.

The model may assign meaning to extra fields, mix ID spaces, treat debug data as evidence, or base a decision on metadata that should not matter.

### Separate semantic and technical IDs

Send an ID to the LLM only when the model must choose an existing object from an explicit allowed list.

Do not send temporary workflow IDs when they are only needed by C# code to connect the response back to current state. If the model chooses an existing object, the ID must come from the candidate list provided to the model.

Allowed:

```json
{
  "searchHits": [
    {
      "entryId": "67e36893e80d",
      "name": "Ponchik"
    }
  ]
}
```

Usually wrong:

```json
{
  "candidate": {
    "candidateId": "tmp-42",
    "name": "Ponchik"
  }
}
```

### Keep output minimal

The output contract should contain the decision and only the values the caller cannot safely reconstruct.

For a new character, the model does not create the new character ID:

```json
{
  "decision": "new",
  "reason": "No matching archive entry found."
}
```

For an existing character, returning the selected entry ID is meaningful:

```json
{
  "decision": "existing",
  "entryId": "67e36893e80d",
  "reason": "Name and aliases match."
}
```

`reason` is acceptable for audit, diagnostics, or human review. Pipeline logic must not depend on prose text when the decision already has structured fields.

### Project internal state into LLM contracts

Use an explicit flow:

```text
Internal model -> LLM input contract -> LLM output contract -> Internal result
```

Bad:

```json
{
  "candidate": {
    "candidateId": "tmp-42",
    "createdAt": "2026-05-28T10:15:00Z",
    "sourceBatch": 17,
    "name": "Ponchik",
    "aliases": ["Ponchika"],
    "debug": {
      "durationMs": 1280
    }
  }
}
```

Good:

```json
{
  "candidate": {
    "name": "Ponchik",
    "aliases": ["Ponchika"],
    "evidence": [
      {
        "pointer": "1.1.1.p4",
        "text": "..."
      }
    ]
  }
}
```

### Treat nullable as meaningful

Nullable C# properties in LLM contracts must not mean "this field is sometimes not useful". If a field is not useful for the model's decision, remove it from the contract.

Nullable is acceptable when the value is genuinely unknown, not applicable for a decision branch, or present only for a specific branch:

```csharp
public sealed record CharacterMatchDecision(
    MatchDecision Decision,
    string? EntryId,
    string Reason);
```

Every nullable field must have a clear null meaning. Do not use nullable because the field might be useful later, because an internal model already has it, or because workflow code needs it outside the prompt.

When a contract is a C# type used through Microsoft Agent Framework / Microsoft.Extensions.AI structured outputs, the spec should not require manually writing the nullable JSON Schema shape. The schema should be generated from the typed contract where the framework supports that path.

### Avoid nullable bool for domain decisions

Use `bool` only for a real binary value that is always true or false.

Avoid `bool?` when `null` is a third domain state. The contract should name that state with an enum.

Bad:

```csharp
public bool? IsSameCharacter { get; init; }
```

Good:

```csharp
public enum SameCharacterDecision
{
    Same,
    Different,
    NotEnoughEvidence
}

public sealed record CharacterIdentityDecision(
    SameCharacterDecision Decision,
    string Reason);
```

Prefer domain-specific enum values such as `Supported`, `Contradicted`, and `NotMentioned` over generic `Yes`, `No`, and `Unknown`.

### Keep schemas strict

LLM contracts should work with structured outputs or strict tool use when the provider and current agent type support it.

Rules:

- use enums for closed decision sets;
- do not use free-form strings for finite options;
- keep field names short, clear, and stable;
- describe important fields through XML comments or description attributes when schema generation uses them;
- do not add fields only to make schema generation easier;
- do not require manual JSON Schema when a typed C# contract is the source of truth and the framework generates the schema.

Example:

```csharp
public enum ResolverDecision
{
    New,
    Existing,
    Merge,
    Split,
    Ignore
}

public sealed record CharacterResolverOutput(
    ResolverDecision Decision,
    string? EntryId,
    string Reason);
```

`EntryId` is nullable because it is needed only for `Existing`. If a result accumulates many branch-specific nullable fields, split the result into clearer forms or redesign the contract.

## Non-goals

- Do not rewrite existing agents only because this rule document exists.
- Do not change domain models.
- Do not change persistence models.
- Do not change workflow orchestration.
- Do not add runtime tests.
- Do not require hand-written JSON Schema when the current agent uses typed C# structured outputs through Microsoft Agent Framework / Microsoft.Extensions.AI.
- Do not change existing prompts unless a future contract change requires it.

## Usage

Use this document as review guidance for every new or changed LLM contract.

Apply the rules incrementally when agent contracts are added or refactored. Existing contracts do not need to be rewritten immediately unless they are already being changed for related work.

Stop and split the work if applying these rules would require unrelated changes to runtime agent behavior, domain models, persistence models, or workflow orchestration.

## Done criteria for contract changes

- LLM input contains only fields needed for the agent decision.
- LLM output contains only the decision and values code cannot safely reconstruct.
- Workflow, debug, UI, persistence, and audit metadata stay outside the prompt unless the model must reason about them.
- Technical IDs stay outside the prompt unless the model must choose from an explicit allowed list.
- Internal models are projected into separate LLM-facing contracts.
- Finite decision sets use enums instead of free-form strings.
- Nullable fields have a clear branch or absence meaning.
- Domain-level third states use named enums instead of `bool?`.
- The contract remains compatible with typed structured outputs for the selected Agent Framework path.

## Review checklist

- Does each field help the model make the current decision?
- Is the field free of workflow, debug, UI, and persistence metadata?
- Does the model need this ID, or only the caller code?
- If the model needs an ID, is it selected from an explicit allowed list?
- Must the model return this field, or can code reconstruct it?
- Can the contract be flatter without losing meaning?
- Are there no fields added for future use?
- Is the contract a projection instead of a full internal object?
- Is every finite decision represented as an enum instead of a free-form string?
- Is there no `bool?` where the third state should be named with an enum?
- Is the meaning of each nullable field clear?
- Is there at least one bad payload and one good payload example?
- Does pipeline logic avoid depending on prose fields such as `reason`?
