# 0012 SPEC JSON character bible UI

Type: spec

Replaces:
- none

Related:
- 0007
- 0008

## Goal

Make the Character Bible canonical storage a structured JSON companion file and replace the Markdown/YAML editor page with a searchable card-based character UI.

## Context

The current Web UI persists a generated Markdown companion file with a fenced YAML block and shows character dossiers as Markdown/YAML. The domain model and `CharacterDossierService` already store structured dossiers and already support JSON serialization. The UI should use that existing service path directly instead of creating another model.

## Scope

- Save and load `<book-name>-character-bible.json` as the canonical companion file.
- Load the old `<book-name>-character-bible.md` fenced YAML block only when JSON is missing.
- Keep the existing `CharacterDossierService` and dossier records as the schema source.
- Add a focused workspace save method for the character bible file.
- Replace the Character Bible page with MudBlazor cards, search, add, edit, delete, reload, and save actions.
- Add a dialog component for structured character editing.
- Update tests for JSON persistence, legacy Markdown/YAML migration, JSON round-trip, and filtering if filtering is extracted.

## Non-goals

- Do not add multi-book workspace support.
- Do not introduce a second character bible model.
- Do not keep Markdown or YAML as canonical persistence.
- Do not parse arbitrary Markdown prose into dossiers.
- Do not redesign the Book or Bible Operations pages.
- Do not add vector search, embeddings, authentication, or operation history.

## Plan

1. Inspect the current dossier model, file store, workspace state, Character Bible page, and tests.
2. Change the file store to JSON persistence with isolated legacy Markdown/YAML loading.
3. Add a focused `SaveCharacterBibleAsync` method to the workspace state.
4. Replace the Character Bible page with cards and client-side filtering.
5. Add the create/edit dialog and delete confirmation.
6. Update tests and run repository verification.

## Pause points

- Stop if MudBlazor dialogs are not configured and enabling them requires broad unrelated changes.
- Stop if the current workspace lifetime prevents dialog edits from using the same `CharacterDossierService`.
- Stop if JSON serialization requires breaking the existing dossier contract.
- Stop if legacy loading cannot stay isolated to the fenced YAML block.

## Acceptance criteria

- The companion path for `novel.md` is `novel-character-bible.json`.
- Character bible data is saved and loaded through `CharacterDossierService.SaveToJson` and `LoadFromJson`.
- Legacy Markdown companions load only when JSON is missing and are not written anymore.
- The Character Bible page shows searchable cards instead of Markdown/YAML tabs.
- Create, edit, and delete update the existing dossier service and save the JSON companion when a book path is available.
- Existing non-manual tests pass.

## Verification

Commands or manual checks:

## Outcome

The JSON companion format is now written directly from `CharacterDossierService.SaveToJson()` using camelCase property names that match the dossier schema. `LoadFromJson()` remains case-insensitive, so existing PascalCase JSON produced by earlier local builds can still be read.

The Web file store writes only `<book-name>-character-bible.json`. It still loads the old Markdown companion's fenced YAML block when JSON is missing, but that legacy path is read-only.

Verification completed:

```bash
dotnet test --filter "Category!=Manual"
dotnet build AiTextEditor.sln
```

Local Web smoke check returned HTTP 200 for `/character-bible` and Browser inspection found the card page title, search, add, save, and empty state controls.
