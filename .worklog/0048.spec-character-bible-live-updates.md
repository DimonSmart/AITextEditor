## Goal

Update the `/character-bible` page while background character bible generation adds or changes dossiers, without refreshing the browser page or resetting the user's current UI state.

## Context

Background character bible generation can continue while the user navigates in the Blazor app. The Character Bible page currently loads the dossier list initially, but users can still see stale cards until they manually refresh after the workflow changes the underlying dossier service.

## Scope

- Notify the open Character Bible page when the workspace's character dossiers change during background generation.
- Update the page's character data source in place and recompute the currently filtered and sorted view.
- Preserve search text, active filters, sorting, selected or expanded character state, and the current loaded book/document state.
- Use stable `CharacterId` values when applying updates so existing cards update in place and duplicates are not created.
- Resynchronize from the current workspace state after a temporary live update failure or reconnect-style notification gap.
- Keep the existing initial page load behavior.

## Non-goals

- Do not periodically reload the whole browser page.
- Do not reset the page ViewModel to apply live updates.
- Do not add multi-user or cross-process synchronization.
- Do not add durable operation history.
- Do not change the character bible JSON storage format.
- Do not change extraction, identity resolution, or profile patching semantics.

## Plan

1. Inspect the current Character Bible page, workspace state notifications, and operation runner commit path.
2. Add or adjust dossier-change notifications so generation publishes committed dossier changes.
3. Make the page subscribe, merge current dossiers by `CharacterId`, and recompute filtered output without resetting UI state.
4. Add focused tests for notification and in-place merge behavior where the existing test surface supports it.
5. Run repository verification.

## Pause points

- Stop if live updates require changing the Blazor circuit lifetime or adding a new transport outside the existing app architecture.
- Stop if per-character updates cannot be observed without a broad rewrite of the workflow commit model.

## Acceptance criteria

- A newly created character appears on `/character-bible` without a browser refresh.
- An existing character card updates without a browser refresh.
- Active filters remain active after updates.
- New characters appear only when they match the active filters.
- Multiple updates for the same `CharacterId` leave one card with the latest state.
- Initial loading still reads the current character bible as before.
- If a live notification is missed or the channel reconnects, the page can resync from the current workspace state without resetting filters.

## Verification

Commands or manual checks:

## Outcome

Implementation uses the existing scoped Blazor workspace notification path. The workflow progress stream can carry a `DossiersSnapshot`; the Web operation runner applies that snapshot to `EditorWorkspaceState`, and the Character Bible page recomputes its current view from the updated data source.

Snapshot granularity follows the current staged workflow: one snapshot after candidate resolution and one after profile patching. It does not emit a separate UI update for every individual candidate or profile tool call.
