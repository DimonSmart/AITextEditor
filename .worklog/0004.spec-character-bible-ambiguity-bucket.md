# 0004 SPEC Character bible ambiguity bucket

Type: spec

Replaces:
- none

Related:
- 0001
- 0003

## Goal

Prevent ambiguous character matches from being auto-committed as new or merged dossiers during character bible workflow runs.

## Context

The staged MAF workflow already has decision contracts for `Ambiguous` and `Defer`, but the resolver currently emits only `Existing` and `New`. The next safety step is to keep uncertain matches visible in the workflow result and leave canonical dossiers unchanged for those candidates.

## Scope

- Detect ambiguous deterministic matches in the current dossier index.
- Emit `CharacterBibleDecisionKind.Ambiguous` decisions with candidate dossier ids.
- Do not create a new dossier for an ambiguous candidate.
- Expose resolver decisions and ambiguity counts in the workflow result.
- Add focused tests for ambiguous matching behavior.

## Non-goals

- Do not add human review UI yet.
- Do not persist ambiguity buckets to Markdown yet.
- Do not add LLM-based merge arbitration yet.
- Do not add embeddings or global consolidation yet.

## Plan

1. Extend workflow result contracts with decisions and ambiguity count.
2. Update deterministic dossier matching to distinguish no-match from ambiguous-match.
3. Skip commit mutations for ambiguous candidates.
4. Add a unit test with two existing candidates sharing the same alias key.

## Pause points

- Stop if ambiguity detection requires changing public dossier storage contracts.

## Acceptance criteria

- Ambiguous candidates are reported as `Ambiguous`.
- Ambiguous candidates do not create new dossiers.
- Existing and new match behavior remains covered by tests.
- Repo checks pass.

## Verification

Commands or manual checks:

```bash
dotnet build AiTextEditor.sln
dotnet test --filter "Category!=Manual"
```

## Outcome

Implemented deterministic ambiguity reporting for the staged character bible workflow.

When a candidate matches multiple existing dossiers with the same best score, the resolver now emits `CharacterBibleDecisionKind.Ambiguous`, includes the candidate dossier ids, and does not create or merge a canonical dossier for that candidate. Workflow results now expose the full resolver decision list and an ambiguity count.

The current ambiguity bucket is in-memory workflow output only. Persisting unresolved decisions into canonical Markdown or a sidecar index remains follow-up work.
