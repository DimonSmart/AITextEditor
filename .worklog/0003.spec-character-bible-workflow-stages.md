Type: spec

## Goal

Split the first MAF character bible workflow into explicit typed stages so traversal, extraction, matching, and commit are visible in the workflow graph.

## Context

The first MAF migration step added a real `WorkflowBuilder` runner, but it still delegates all work to `CharacterDossiersGenerator` inside one executor. The next step should expose the pipeline shape described in the roadmap while preserving current behavior.

## Scope

- Add typed contracts for character extraction proposals, matching proposals, resolver decisions, and commit batches.
- Extract reusable character extraction and deterministic dossier matching logic from the current generator path.
- Build a MAF graph with separate executors for traversal/extraction, matching/resolution, and commit.
- Keep the existing `CharacterDossierService` as the only write path.
- Preserve existing plugin behavior.
- Add focused unit tests for the staged workflow.

## Non-goals

- Do not add canonical Markdown persistence yet.
- Do not add embeddings, vector search, or global consolidation yet.
- Do not add human approval flow yet.
- Do not attempt literary pronoun/coreference resolution beyond the current extractor behavior.
- Do not replace Semantic Kernel chat completion in this change.

## Plan

1. Introduce intermediate workflow contract records.
2. Move extraction/matching/commit logic behind small services or internal executors.
3. Change `CharacterBibleWorkflowRunner` from one executor to a staged graph.
4. Keep `CharacterDossiersGenerator` behavior compatible by delegating to the same staged services where practical.
5. Extend tests to assert staged generation and refresh behavior.

## Pause points

- Stop if preserving existing generator behavior requires a broad rewrite of private matching logic.

## Acceptance criteria

- The MAF workflow has more than one executor and passes typed messages between stages.
- The workflow still produces the same dossier output shape as before.
- Ambiguous/deferred decisions are represented in contracts even if the first implementation commits only existing/new decisions.
- Existing non-manual tests pass.

## Outcome

Implemented the staged MAF workflow while preserving the existing extraction and deterministic merge behavior.

The workflow now passes typed messages through four executors:

1. collect Markdown paragraphs from the current document or changed pointers;
2. extract character candidates with the existing LLM prompt;
3. resolve candidates into existing/new commit decisions against the current dossier index;
4. commit the projected dossier set through `CharacterDossierService`.

The new contracts include `CharacterBibleDecisionKind.Ambiguous` and `CharacterBibleDecisionKind.Defer`, but the current resolver still emits only `Existing` and `New` to match the previous auto-commit behavior. Ambiguous handling remains a follow-up.

Verification:

```bash
dotnet test --filter "Category!=Manual"
dotnet build AiTextEditor.sln
```

Manual functional check on `neznayka_sample.md` after removing the test-side fallback:

```bash
dotnet test tests/AiTextEditor.Tests/AiTextEditor.Tests.csproj --filter "FullyQualifiedName~McpFunctionalTests.CharacterDossiersCommandScenarios_ReturnExpectedCatalog"
```

Result: the command path passed through `SemanticKernelEngine`, invoked `character_dossiers.generate_character_dossiers`, then `character_dossiers.get_character_dossiers`, and returned JSON without direct generator fallback. The workflow collected 25 paragraphs, extracted 9 candidates, resolved 9 decisions, and committed version 2.

Limitation found: one extraction batch returned empty assistant `content` while its reasoning listed `Знайка`, `Фуксия`, `Селёдочка`, and `Стекляшкин`. Because extraction currently treats empty/non-JSON content as an empty candidate list, the final catalog had 8 characters and missed at least `Фуксия`, `Селёдочка`, and `Стекляшкин`. Follow-up: make extractor output validation explicit and surface invalid/empty batch outputs as deferred or failed decisions instead of silently losing candidates.

Repeating the same manual check with `LLM_MODEL=gpt-oss:120b-cloud` produced valid JSON `content` for each extraction batch. The workflow collected the same 25 paragraphs, extracted 13 candidates, resolved 13 decisions, and produced 11 final characters, including `Фуксия`, `Селёдочка`, and `Стекляшкин`. This points to a model/output-compliance failure on `gpt-oss:20b-cloud`, but the product issue remains ours: extraction must not treat a reasoning-only or empty-content model response as a successful empty extraction.
