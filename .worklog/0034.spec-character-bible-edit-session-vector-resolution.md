## Goal

Make Character Bible generation resolve and patch candidates against one mutable in-memory catalog session, then connect identity resolution search to the current session through snapshot-based vector search.

## Context

The current workflow still carries projected dossiers through a commit plan and applies the final result through a committer. Identity resolution also searches through the old archive search service. This means a resolver call can miss characters created earlier in the same run unless the projected archive is rebuilt and passed around.

The desired workflow is direct:

```text
collect -> extract -> resolve_and_update_catalog -> patch_catalog -> return_result
```

The resolver, patcher, and search tool must all observe the same current in-memory `CharacterDossiers` snapshot.

## Scope

- Add `CharacterDossierEditSession` that owns a deep copy of the starting `CharacterDossiers` and tracks decisions and changes.
- Replace `CharacterBibleCommitPlan` as the active workflow state with a run state that carries the edit session.
- Apply resolver decisions directly to the edit session after each candidate.
- Remove active use of commit operations and `CharacterBibleCommitter`.
- Make workflow finish by returning `session.Current` without saving to disk.
- Change `ICharacterVectorSearchTool.SearchAsync` to accept an explicit `CharacterDossiers` snapshot.
- Remove `CharacterDossierService` from `CharacterVectorSearchTool`.
- Route `CharacterBibleResolver` search through `ICharacterVectorSearchTool` using the current session snapshot.
- Delete the old lexical `CharacterArchiveSearchService` path and related unused request/hit models.
- Add focused tests for session mutation, sequential visibility, snapshot vector search, workflow output, and no fallback.

## Non-goals

- Do not add undo UI.
- Do not autosave during workflow execution.
- Do not save generated dossiers from the workflow.
- Do not keep a feature flag, compatibility overload, or lexical fallback.
- Do not rewrite patch proposal into an `actions[]` contract.
- Do not keep two active commit or search paths.

## Plan

1. Add `CharacterDossierEditSession` and tests for catalog mutation operations.
2. Replace commit-plan workflow state with a run state and direct decision applier.
3. Update workflow executors and generator to return `session.Current`.
4. Make vector search snapshot-based and remove service-backed catalog reads.
5. Inject vector search into identity resolution and keep the LLM-facing `search_characters(query, limit)` tool.
6. Delete old archive search and commit artifacts from active code.
7. Run repository checks with manual/local LLM tests excluded by local instructions.

## Pause points

- Stop if deleting `CharacterBibleCommitPlan` requires rewriting the patch proposal contract.
- Stop if `CharacterDossierService` APIs force persistence during workflow generation.
- Stop if snapshot vector indexing requires a broad UI search rewrite.
- Stop if split proposal code blocks deleting old archive search and needs a separate product decision.
- Stop if mutating `session.Current` has hidden persistence side effects.

## Acceptance criteria

- Workflow no longer uses `ProjectedDossiers`, `CharacterBibleCommitPlan`, commit operations, or `CharacterBibleCommitter` in the active path.
- The workflow mutates one `CharacterDossierEditSession` and does not write dossiers to disk during generation.
- Candidate resolution is sequential and candidate N+1 sees catalog changes from candidate N.
- `CharacterBibleWorkflowOutput` returns the updated `session.Current`.
- `ICharacterVectorSearchTool.SearchAsync` accepts a `CharacterDossiers` snapshot and has no old overload.
- `CharacterVectorSearchTool` indexes the provided snapshot and no longer depends on `CharacterDossierService`.
- Identity resolver uses vector search through the unchanged LLM-facing `search_characters(query, limit)` tool.
- `CharacterArchiveSearchService` and old lexical search references are removed.
- Vector search failures are visible and are not hidden by fallback logic.

## Outcome

Implemented the direct in-memory catalog session and snapshot-based vector resolution path.

Notable implementation details:

- `CharacterDossierEditSession` owns a copied `CharacterDossiers` snapshot and applies candidate creation, alias/gender/profile changes, evidence index entries, suspect entries, identity conflicts, importance levels, and audit entries in memory.
- `CharacterBibleRunState` replaced the commit plan as the active workflow carrier.
- Resolver decisions are applied immediately, so later candidates and resolver search calls see earlier catalog mutations from the same run.
- The finish workflow stage returns `session.Current` and does not call `CharacterDossierService.ReplaceDossiers`.
- `ICharacterVectorSearchTool.SearchAsync` now accepts an explicit `CharacterDossiers` snapshot.
- `CharacterVectorSearchTool` no longer depends on `CharacterDossierService`.
- `CharacterArchiveSearchService`, `CharacterBibleCommitter`, commit operations, and projection index were removed.
