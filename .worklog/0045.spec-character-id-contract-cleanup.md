## Goal

Make numeric `CharacterId` the only persisted and LLM-visible stable character identifier in the Character Bible pipeline.

## Context

The integer `CharacterId` model exists, but old `entryId` terminology remains in resolver DTOs, search results, prompts, diagnostics, and tests. New-character resolution also still risks mixing the identity decision stage with the archive creation stage. Diagnostics need to show `nextCharacterId` so runs can prove IDs come from persisted archive state instead of character count.

## Scope

- Replace character-related `entryId` and `entryIds` fields in Character Bible resolver/search DTOs, prompts, logs, and tests with `characterId` and `characterIds`.
- Keep resolver search results visible to the LLM with real numeric `characterId` values.
- Keep `new` identity decisions without a `CharacterId`; create IDs only when archive/edit-session code creates the dossier.
- Log `nextCharacterId` when archives are loaded, characters are created, archives are saved, and the workflow finishes.
- Load old Character Bible JSON without `nextCharacterId` by deriving it as `max(characterId) + 1`, or `1` for an empty archive.
- Ensure profile update diagnostics for a specific character include `characterId`; keep names only as diagnostic context.
- Keep profile update tools character-scoped and do not expose `characterId` or `character` as LLM tool arguments.
- Add focused tests for terminology, new/existing decisions, `nextCharacterId` persistence, ID reuse prevention, and profile update logs.

## Non-goals

- Do not hide real archive IDs from the resolver LLM behind `rank`, `matchKey`, or temporary aliases.
- Do not add compatibility wrappers or duplicate runtime APIs beyond the explicitly required missing-`nextCharacterId` JSON migration.
- Do not change canonical name, alias, vector scoring, or identity heuristics.
- Do not expose character IDs as profile update tool parameters.

## Plan

1. Rename Character Bible resolver/search contracts and prompt fields from `entryId` terminology to `characterId`.
2. Move ID issuance fully into archive/edit-session creation paths and adjust resolver decision handling.
3. Update diagnostic logging for resolver/search/profile/workflow/archive events.
4. Add missing-`nextCharacterId` JSON migration in the archive load path.
5. Update focused tests and run repository checks with manual local-LLM tests excluded.

## Pause points

- Stop if renaming `EntryId` in vector-search internals would require changing a non-character third-party API.
- Stop if old JSON migration requires accepting non-numeric character IDs.

## Acceptance criteria

- Character Bible domain models, internal DTOs, LLM-facing DTOs, prompts, and logs do not use `entryId` or `entryIds` for characters.
- Resolver search results expose real numeric `characterId` values.
- `new` resolver decisions have `CharacterId == null` and empty `CharacterIds`.
- Created characters receive IDs only from archive/edit-session creation state and advance `NextCharacterId`.
- Run logs show `nextCharacterId` on archive load, character creation, archive save, and workflow finish.
- Missing-`nextCharacterId` JSON loads with `NextCharacterId = max(characterId) + 1`, or `1` when empty.
- Character-specific `profile.update.*` events include `characterId`.
- Repo checks pass with manual local-LLM tests excluded.
