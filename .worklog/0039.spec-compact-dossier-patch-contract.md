## Goal

Make `patch.proposal.call` and its review stage use compact LLM-facing contracts that expose only target name, current profile text, and one materialized evidence list, then apply returned profile facts as atomic additions.

## Context

The current dossier patch proposal prompt resembles an internal workflow DTO. It exposes technical candidate and dossier IDs, identity-resolution details, duplicate evidence shapes, aliases, and a partial `profilePatch` object. That gives the model too much implementation detail and encourages it to replace profile sections instead of proposing small evidence-backed additions.

The patch stage runs after identity resolution. It should not re-evaluate identity and should not see workflow correlation fields. It should decide whether the current evidence adds stable profile facts for the already resolved character.

## Scope

- Add explicit prompt input records for `patch.proposal.call` with `target.name`, `currentProfile`, and `newEvidence[] { pointer, text }`.
- Keep technical IDs and identity-resolution data available to code and diagnostics, but outside the prompt payload.
- Replace `profilePatch` output with `additions[]` where every addition has a profile field, text, and evidence pointers.
- Update the patch prompt to forbid full field rewrites and unsupported inference.
- Apply additions by appending to profile fields, never by replacing existing non-empty fields.
- Reject or normalize invalid proposals before apply, including missing evidence pointers and pointers not present in `newEvidence`.
- Update `DossierConsistencyReviewerAgent` input to use the same compact target/profile/proposal/evidence shape.
- Change reviewer issues to structured issue codes.
- Add focused tests for prompt JSON shape, single evidence representation, additive apply, no-useful-changes, and unsupported-claim review behavior where practical.

## Non-goals

- Do not change identity resolution behavior.
- Do not change `extract_character_bible_candidates`.
- Do not change Character Bible dossier storage unless needed to append additions.
- Do not move alias handling into the patch stage.
- Do not change the number or boundaries of patch calls in this slice.
- Do not implement full character-level grouping or bounded regrouping as part of this change.

## Plan

1. Locate current patch proposal and review DTOs, prompt builders, prompts, and apply code.
2. Introduce compact LLM-facing DTOs and project existing internal state into them immediately before model calls.
3. Update prompt Markdown and structured output contracts.
4. Change apply validation and merge behavior from partial-profile replacement to atomic additions.
5. Update reviewer input/output and tests.

## Pause points

- Stop if `profilePatch` is consumed in many downstream places and replacing it with additions requires a broad migration.
- Stop if aliases are currently added only through `patch.proposal.call` and cannot safely be moved or left out.
- Stop if downstream consumers depend on old `reason` or `changedFields` fields.
- Stop if prompt DTOs cannot be separated from internal DTOs without a large refactor.

## Acceptance criteria

- `patch.proposal.call` prompt JSON contains only `target.name`, `currentProfile`, and `newEvidence[] { pointer, text }`.
- The patch proposal prompt JSON does not contain `candidateId`, `characterId`, `identityDecision`, `candidateIds`, separate `pointers`, duplicate `evidence`/`evidenceContexts`, or aliases.
- Patch proposal output contains `status` and `additions[]`, not `profilePatch`.
- Every addition contains `field`, `text`, and `evidencePointers`.
- Apply appends approved additions and does not replace existing non-empty profile fields.
- Reviewer receives compact target/profile/proposal/evidence input and structured issues.
- Evidence with only a bare character mention can result in `noUsefulChanges` and no apply.
