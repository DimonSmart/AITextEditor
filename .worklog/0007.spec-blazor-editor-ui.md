## Goal

Add a Blazor UI for loading and editing a Markdown book, inspecting the existing linear document model, viewing structured character dossiers as Markdown/YAML, and running the existing character bible workflow with visible operation progress.

## Context

The repository is currently an MCP-oriented toolkit around `EditorSession`, `MarkdownDocumentRepository`, the linear document model, and `CharacterDossierService`. The character bible workflow already exists in `AiTextEditor.Agent` and writes through `CharacterDossierService`.

The UI must expose those existing services instead of introducing a separate book model or a second character bible store. Character bible Markdown is a projection over structured dossiers for this change because canonical Markdown persistence is still out of scope.

## Scope

- Add `src/AiTextEditor.Web` as a Blazor app targeting the repository framework.
- Add left navigation for Book, Linear Items, Character Bible, and Bible Operations.
- Add scoped UI workspace state around a single `EditorSession`.
- Add a reusable split Markdown editor/preview component.
- Implement Book editing through `EditorSession.LoadDefaultDocument`.
- Implement Linear Items inspection and simple filters.
- Render character dossiers to Markdown and expose YAML edit/apply through `CharacterDossierService`.
- Add an in-memory character bible operation model, command parser, and UI-facing operation runner around `CharacterBibleWorkflowRunner`.
- Show coarse progress events, final summaries, failures, and cancellation in the operations page.
- Add focused tests for renderer, parser, and runner behavior where practical.

## Non-goals

- Do not add a full conversational LLM chat.
- Do not add persistent operation history.
- Do not add multi-book workspace support.
- Do not add Git/versioning UI.
- Do not parse arbitrary character bible Markdown back into dossiers.
- Do not implement canonical character bible Markdown persistence.
- Do not add human approval workflow for ambiguous candidates.
- Do not add vector search, embeddings, target set editing, authentication, or multi-user state isolation.

## Plan

1. Inspect existing project APIs and workflow construction dependencies.
2. Add the Blazor Web project and solution entry, stopping if MatBlazor is incompatible with the selected target framework.
3. Add UI services and models that wrap existing core and agent services.
4. Implement the four pages and shared Markdown component.
5. Add focused tests for pure UI services and operation runner behavior.
6. Run build, non-manual tests, and start the Web project.

## Pause points

- Stop and report the exact conflict if MatBlazor cannot be added cleanly for the selected target framework.
- Stop before replacing the existing workflow or dossier write path with UI-owned logic.

## Acceptance criteria

- `AiTextEditor.Web` exists and is included in `AiTextEditor.sln`.
- The app starts as a Blazor app with Book, Linear Items, Character Bible, and Bible Operations navigation.
- Book edits update Markdown preview and the existing `EditorSession`.
- Linear Items shows items produced by the existing parser with basic filters.
- Character Bible shows dossier Markdown projection and supports raw YAML apply.
- Bible Operations runs the existing character bible workflow through an operation runner with visible events, summary, cancellation, and failure display.
- Existing non-manual tests still pass.

## Outcome

Implementation kept the UI state in a scoped `EditorWorkspaceState` around one `EditorSession`. Character bible Markdown is a Web-layer projection over `CharacterDossiers`; YAML remains the only editable dossier representation in this UI.

The operation page uses coarse UI events around the existing `CharacterBibleWorkflowRunner`. The production workflow client validates `LLM_BASE_URL`, `LLM_API_KEY`, and `LLM_MODEL` only when an AI operation is run, so the non-LLM pages can still work without model configuration.

MatBlazor 2.10.0 restored as compatible with the `net9.0` Web project. The UI uses MatBlazor buttons where practical and keeps the rest as ordinary Blazor/Bootstrap markup.

Verification completed:

```bash
dotnet build AiTextEditor.sln
dotnet test --filter "Category!=Manual"
```

Manual HTTP smoke checks returned 200 for `/book`, `/items`, `/character-bible`, and `/bible-operations` on `http://localhost:5109`. Browser screenshot verification was not completed because the browser automation plugin had no authorized page to attach to in this session.
