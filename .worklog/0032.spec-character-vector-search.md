## Goal

Add a pure semantic vector search tool over current Character Bible dossiers.

## Context

Consumers need to ask which characters are semantically similar to free text, even when the query does not name the character. Existing archive search is deterministic identity/name matching and must stay separate from semantic retrieval.

## Scope

- Add `ICharacterVectorSearchTool` and compact result contracts.
- Build an in-memory vector index from current `CharacterDossierService` characters.
- Keep the index automatically synchronized before each search.
- Add an explicit semantic search entry point to the Character Bible card page.
- Apply semantic ordering before the existing card filters when semantic search is active.
- Use a dedicated Ollama embedding model, defaulting to `bge-m3:latest`.
- Keep embedding model selection as construction/options state, not a per-call embedding parameter.
- Compute fingerprints from schema version, embedding model id, and normalized indexed text.
- Score only by cosine similarity and sort ties by name and entry id.

## Non-goals

- Do not add exact name, alias, normalized, or fuzzy matching.
- Do not add name boosts, identity resolution, character merge/split, or hybrid search.
- Do not expose full dossiers, evidence graph, patches, or embedding vectors.
- Do not add persistent vector storage, vector databases, ANN/HNSW, or cross-book search.
- Do not stabilize current profile fields as the public search contract.

## Plan

1. Add vector search contracts and in-memory index implementation in the agent character bible area.
2. Add an internal embedding client boundary and an Ollama `/api/embed` Web implementation.
3. Add settings for the selected embedding model and register the scoped search tool.
4. Add a short semantic-search control to the Character Bible card page and keep ordinary filters as a post-filtering step.
5. Cover indexing, fingerprinting, sorting, card shape, and no-name-boost behavior with fake embedding tests.

## Pause points

- Stop if the implementation requires changing the public Character Bible storage model.

## Acceptance criteria

- `SearchAsync` validates empty query/limit and returns empty results for empty indexes.
- Index synchronization is automatic and handles new, changed, unchanged, and removed characters.
- Fingerprints change when indexed text, schema version, or embedding model id changes.
- Score is cosine similarity only.
- The Character Bible page can order cards by semantic search first, then apply the existing filters.
- Public results contain only `EntryId`, `Name`, `Gender`, `Aliases`, `Summary`, and `Score`.
- Repo checks pass with manual/local LLM tests excluded.
