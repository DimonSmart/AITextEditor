## Goal

Move the character bible model-call path away from Semantic Kernel and onto the Microsoft Agent Framework migration path with explicit typed model contracts.

## Context

The staged MAF workflow already owns traversal, extraction, resolution, and commit orchestration, but the extraction executor still depends on Semantic Kernel `IChatCompletionService` and a text JSON response convention. That violates the project rule that model calls must be typed through tool contracts, response-format schemas, or equivalent structured contracts.

The immediate target started as the character bible path. During implementation the same typed-model requirement also forced the remaining command-agent and cursor-agent model surfaces off Semantic Kernel so the package could be removed instead of partially bypassed.

## Scope

- Fix the existing functional-test blocker caused by the new typed extraction response contract.
- Introduce project-owned typed chat completion abstractions for character extraction and cursor-agent structured calls.
- Implement OpenAI-compatible typed chat completion clients without Semantic Kernel.
- Wire `CharacterDossiersGenerator` to the new abstraction.
- Replace the old Semantic Kernel command engine with an agentic workflow engine and normal service calls.
- Remove Semantic Kernel attributes, filters, chat-history helpers, and package reference.
- Keep the MAF workflow graph and deterministic executors as the orchestration layer.
- Keep extraction output fail-fast on empty or contract-invalid model responses.
- Update focused tests to assert typed response-format usage.

## Non-goals

- Do not add compatibility fallbacks for old JSON-array extraction responses.
- Do not change dossier storage contracts.
- Do not add embeddings, vector search, or global consolidation here.

## Plan

1. Update the CP1251 functional-test stub to return the typed extraction response object.
2. Add typed model client contracts for character extraction and structured cursor-agent calls.
3. Add OpenAI-compatible implementations that send JSON schema response format directly over HTTP.
4. Replace `IChatCompletionService` dependency in `CharacterDossiersGenerator`.
5. Replace the Semantic Kernel command engine with `AgenticWorkflowEngine`.
6. Adapt tests and run build/non-manual tests.

## Pause points

- Stop if the local OpenAI-compatible server rejects JSON schema `response_format`, because that changes the provider capability assumptions.

## Acceptance criteria

- Character bible extraction no longer depends on Semantic Kernel chat completion interfaces.
- `src` and tests no longer reference Semantic Kernel types or package references.
- Character extraction uses a typed response contract instead of scraped JSON text.
- Empty model content fails fast and does not produce a successful empty extraction.
- Contract-invalid model content fails fast even when the endpoint ignores JSON schema strictly.
- Existing non-manual tests pass.
- Repository build passes.

## Outcome

Implemented the full removal path rather than leaving a partial bypass:

- Added `AgenticWorkflowEngine` / `AgenticWorkflowContext` and removed `SemanticKernelEngine` / `SemanticKernelContext`.
- Removed `Microsoft.SemanticKernel` from `AiTextEditor.Agent.csproj`.
- Removed SK-specific function attributes, invocation filters, and chat-history compressor.
- Added `OpenAiStructuredChatModelClient`, `OpenAiCharacterExtractionModelClient`, and `OpenAiCursorAgentModelClient`.
- Moved character extraction and cursor-agent calls to typed JSON-schema response-format requests.
- Added runtime contract validation because the local Ollama/OpenAI-compatible endpoint can accept `response_format: json_schema` while still returning fields outside the schema, for example `sentence` instead of `example`.
- Updated tests and functional stubs to use typed clients.
- Improved `CharacterBibleWorkflowRunner` so MAF executor/workflow failures rethrow the original exception instead of hiding it behind `character_bible_workflow_produced_no_result`.
