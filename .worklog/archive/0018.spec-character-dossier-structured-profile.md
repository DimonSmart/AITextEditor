## Goal

Expand Character Bible dossiers from a short description plus legacy generic facts into a stable structured character profile that is useful for understanding and reproducing a character without turning the card into a plot summary.

## Context

`CharacterDossier` currently stores name, aliases, description, gender, importance, and generic `Facts`. The current extraction contract only produces name, gender, aliases, and description, so the UI cannot show a real structured character card. Worklog 0013 already records that generated dossiers still have no facts because facts are not part of the extraction contract.

`Facts` should remain as a legacy/fallback extension area, but the main character card and editor should prefer a typed profile.

## Scope

- Add a `CharacterProfile` model with appearance, background/status/education, psychological profile, speech/communication, and key role bonds.
- Keep `Description`, `Facts`, and `ImportanceLevel`; old JSON and legacy Markdown/YAML imports must still load.
- Normalize profile data on dossier load/save/upsert: trim strings, create an empty profile when missing, remove incomplete role bonds, deduplicate role bonds by character name and role, and keep deterministic ordering.
- Extend the typed extraction response contract and system prompt so generated candidates include Russian structured profile fields.
- Preserve profile data through `CharacterBibleCharacterCandidate`, generator conversion, and dossier merge.
- Merge generated profile sections only into empty existing sections, and append only new role bonds.
- Update Character Bible cards to show compact profile preview and profile-based completeness.
- Update the edit dialog so profile editing is primary and generic facts are a legacy/advanced section.
- Update search and incomplete filtering to use profile fields, key role bonds, and legacy facts.
- Update Markdown projection to render profile sections before additional facts.
- Add focused tests for JSON round-trip, legacy load, normalization, extraction validation, generator merge, search, and Markdown rendering.

## Non-goals

- Do not remove legacy `Facts` in this change.
- Do not migrate old facts into profile fields automatically.
- Do not add relationship graphs, timelines, scene history, embeddings, vector search, or database storage.
- Do not store every relationship, every relative, or every plot event.
- Do not redesign unrelated pages.
- Do not change the companion file name or introduce a second storage file.

## Plan

1. Add the core profile records and normalization rules.
2. Extend extraction DTOs, validation, prompt text, and candidate contracts.
3. Thread profile data through generator accumulation and merge behavior.
4. Update card, dialog, search, and Markdown projection.
5. Add focused tests and run the non-manual solution checks.

## Pause points

- Stop and ask if preserving old files requires changing the canonical JSON file name.
- Stop and ask if generated profile data would overwrite non-empty manual profile sections.
- Stop and ask if removing `Facts` becomes a hard requirement, because current scope keeps it as legacy data.

## Acceptance criteria

- Character dossiers have a non-null normalized `Profile` after load or upsert.
- Old character bible JSON files without `profile` still load.
- New JSON files save and load all profile fields and role bonds.
- Generated dossiers can populate profile fields.
- Manual profile edits are not overwritten during refresh.
- Character cards show compact profile preview and no longer use fact count for completeness.
- The edit dialog allows editing all profile sections and role bonds.
- Search includes profile fields, role bonds, aliases, description, and legacy facts.
- Markdown projection renders the profile sections in stable order and legacy facts under `Additional facts`.
- `dotnet build AiTextEditor.sln` passes.
- `dotnet test --filter "Category!=Manual"` passes.

## Outcome

Implementation keeps `Facts` as legacy data and preserves existing facts during generation refreshes. Generated profile data fills only empty profile sections and appends non-duplicate role bonds.

Follow-up fix: `CharacterExtractionCharacter` must keep a single JSON-supported constructor. A convenience overload for tests broke `System.Text.Json` deserialization of model responses during generation, so the DTO now uses one constructor with an optional `Profile` parameter and has a regression test with a real structured JSON response.


The full reload-from-book and generation/refresh manual smoke path was not run.
