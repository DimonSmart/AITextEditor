## Goal

Use a stable auto-incremented integer `CharacterId` for Character Bible dossiers instead of a name-derived string hash.

## Context

The current archive and in-memory edit session derive dossier IDs from character names. A name is not a stable or unique internal address. New dossier IDs must be issued by archive code when a `new` resolver decision is applied, while existing dossiers keep their numeric IDs.

## Scope

- Change persisted and internal dossier references from string `CharacterId` values to integers.
- Add persisted `NextCharacterId` to the Character Bible root model.
- Issue new IDs from one archive/edit-session creation method and advance `NextCharacterId` without using character count.
- Keep `new` resolver decisions without an ID until the apply stage creates the dossier.
- Keep `existing` resolver decisions addressed by their existing integer ID.
- Keep profile update tools scoped to a known numeric character ID without adding it to LLM tool parameters.
- Preserve multiple name or alias matches instead of treating a name as a unique key.
- Update focused tests and sample persisted JSON used by tests.

## Non-goals

- Do not change canonical-name behavior.
- Do not add a canonicalization stage or deterministic resolver.
- Do not optimize prompts or profile evidence contracts.
- Do not add stored ambiguity flags.
- Do not require unique names or aliases.
- Do not let the LLM create new character IDs.
- Do not keep name-derived string IDs as a runtime compatibility path.
- Do not add migrations, legacy JSON or YAML readers, or fallback loading paths.

## Plan

1. Add numeric archive model fields and strict version 3 JSON loading.
2. Centralize new dossier creation in the edit session and persisted dossier service.
3. Convert resolver, evidence, patching, vector search, UI services, and tests to numeric IDs.
4. Run repository checks with manual local-LLM tests excluded.

## Pause points

- Stop if the existing `Version` field is used as a mutable revision in a way that prevents switching directly to schema version `3`.

## Acceptance criteria

- `CharacterDossier.CharacterId` is an `int`.
- `CharacterDossiers.NextCharacterId` stores the next unused ID.
- New dossiers receive IDs only when archive code applies creation.
- `NextCharacterId` advances monotonically and does not depend on dossier count.
- Existing resolver decisions reuse the selected numeric ID.
- Persisted JSON loading accepts only the current numeric-ID schema.
- Duplicate normalized names or aliases remain representable.
- Profile update tools remain character-scoped and do not expose `CharacterId` as an LLM argument.
- Repo checks pass with manual local-LLM tests excluded.

## Outcome

Implementation notes:

- The previous `Version` field behaved as a mutable revision counter. It now represents storage schema version `3`.
- JSON loading is strict: only schema version `3` with explicit `nextCharacterId` and numeric IDs is accepted.
- The old Markdown/YAML companion fallback and YAML service APIs were removed.
