## Goal

Rename Character Bible character name-form data from aliases to observed name forms, keep observed forms as retrieval signals, and add a separate LLM stage that normalizes a dossier canonical display name from one character card.

## Context

The current `Aliases` fields are used for exact name forms found in text: inflections, titles, address forms, nicknames, and spelling variants. The name is misleading in domain code, LLM prompts, UI, markdown, search, and tests. Canonical name refinement also currently lives in the edit session, but canonical display-name normalization should be an explicit LLM agent stage with a narrow contract.

## Scope

- Rename core fields and methods from `Aliases`/`AliasExamples` to `ObservedNameForms`/`ObservedNameFormExamples`.
- Update extraction DTOs and prompt files to emit `observedNameForms`.
- Write only `observedNameForms` and `observedNameFormExamples`; do not add legacy `aliases`/`aliasExamples` migration or fallback reading.
- Build short observed name form examples in C# from evidence excerpts without trimming full evidence.
- Add the `normalize_character_canonical_name` LLM stage with a prompt in a separate Markdown file.
- Call canonical-name normalization only for newly created dossiers and dossiers that receive new observed name forms.
- Remove active `RefineCanonicalName` production usage.
- Update deterministic name index, vector indexed text, UI labels, markdown export, diagnostics, and focused tests.

## Non-goals

- Do not split aliases and observed name forms into separate entities.
- Do not add a field for true nicknames.
- Do not generate possible inflections.
- Do not add generated name forms to JSON, UI, search, or indexes.
- Do not add final canonical-name audit.
- Do not add complex Russian-language code normalization.
- Do not broaden identity resolver behavior beyond the field rename.
- Do not add permanent failure flags for canonical-name normalization.

## Plan

1. Inspect current model, extraction, edit session, persistence, search, UI, markdown, prompt, and tests.
2. Rename fields, DTOs, methods, labels, and renderer output to observed name forms.
3. Raise the dossier schema version for the renamed persisted fields without adding legacy JSON compatibility.
4. Add short observed name form example post-processing from existing evidence excerpts.
5. Add the canonical-name normalizer agent, response validation, logging, and workflow integration.
6. Remove `RefineCanonicalName` and update affected tests.
7. Run focused tests and repository checks with manual local LLM tests excluded.

## Pause points

- Stop if aliases are exposed as a public API that cannot be renamed directly.
- Stop if the renamed persisted fields require compatibility after all.
- Stop if the normalizer requires changing the shared `AgenticModelClient` contract.
- Stop if removing `RefineCanonicalName` breaks an unexpected non-normalization workflow.
- Stop if UI dependencies on the old alias field are broad enough to need a separate UI decision.

## Acceptance criteria

- Production domain, LLM-facing, UI, markdown, search, and test terminology uses observed name forms for exact forms found in evidence.
- Extraction returns `observedNameForms`, not `aliases`.
- New JSON writes observed-name-form property names and the schema version is raised for the breaking field rename.
- `ObservedNameFormExamples` are short, contain the exact observed form when available, and do not trim `Evidence.Excerpt`.
- Canonical display names are normalized by the dedicated LLM stage and left unchanged on insufficient evidence or invalid responses.
- `RefineCanonicalName` has no production references.
- Search by inflected observed forms still works.

## Outcome

Implementation note:

- The original task text included legacy `aliases` / `aliasExamples` JSON read compatibility, but the active requirement was corrected to no compatibility. The implemented storage model raises the dossier schema version and writes only `observedNameForms` / `observedNameFormExamples`.
- Canonical name normalization is an explicit `normalize_character_canonical_name` workflow stage between identity resolution and profile update. Invalid or insufficient normalizer responses leave the dossier name unchanged.
- Observed name form examples are generated in C# from evidence excerpts and capped to short context snippets; evidence excerpts remain full.
