## Goal

Complete the Character Bible pipeline shape described by the multi-agent architecture plan.

## Context

The previous slices narrowed extraction, added dossier patch proposals, and split deterministic resolver responsibilities. The remaining architecture needs explicit candidate post-processing, reviewer/split contracts, operation-based commit, and Character Bible support collections for suspect archive, evidence, conflicts, and audit.

## Scope

- Add `CandidatePostProcessor`.
- Add explicit identity resolver agent contracts and prompt resource while keeping deterministic identity as the active resolver for this slice.
- Add `DossierConsistencyReviewerAgent` contracts and prompt resource, with a conservative default reviewer path for tests.
- Add `SplitCandidateAgent` contracts and prompt resource; do not apply splits automatically.
- Add operation-based `CharacterBibleCommitPlan` and `CharacterBibleCommitter`.
- Extend Character Bible storage with suspect archive, evidence index, identity conflicts, and audit trail.
- Preserve current successful existing/new generation behavior.
- Verify through the Web UI with a 20-paragraph full scan limit.

## Non-goals

- Do not build a manual review UI in this slice.
- Do not auto-apply split proposals.
- Do not replace deterministic identity resolution with LLM identity resolution as the default behavior.
- Do not add vector search or embeddings.

## Plan

1. Add the remaining contracts/prompts and deterministic/no-op active paths where needed.
2. Route extraction through post-processing.
3. Route patch proposals through reviewer.
4. Commit via operations and record support collections.
5. Run build/tests.
6. Run Web UI and verify book load plus Character Bible generation with 20 paragraph limit.

## Pause points

- Stop if full UI verification requires unavailable local LLM settings or a server that cannot be reached.

## Acceptance criteria

- The codebase has explicit stages matching the multi-agent plan.
- Default workflow still generates/refreshes dossiers.
- Support collections serialize with the Character Bible JSON.
- Manual/local LLM tests remain excluded from standard test verification.
- UI verification is attempted and reported with concrete outcome.

## Outcome

Implemented the remaining multi-agent pipeline shape:

- extraction now routes through `CandidatePostProcessor`;
- identity resolver agent/reviewer/split contracts have separate Markdown prompt resources;
- dossier patches are reviewed before applying;
- commit plans are operation-based and applied by `CharacterBibleCommitter`;
- Character Bible JSON now carries suspect archive, evidence index, identity conflicts, version, and audit trail.

The production resolver now uses `SuspectArchiveResolverAgent` with read-only archive search results. The deterministic resolver remains as the fallback path for tests and failed/invalid identity resolver calls. Splits are represented by contracts only and are not applied automatically.

Final UI verification completed on `2026-05-26` through the Web app after enabling the production identity resolver agent:

- loaded `tests\Samples\neznaika_na_lune_full.md`;
- saved settings with full-scan item limit `20`;
- ran `Generate character bible`;
- operation completed successfully with 20 paragraphs, 13 candidates, 13 decisions, and 0 ambiguous decisions;
- one extraction batch returned malformed JSON, recovery failed, and retry succeeded on attempt 2/3, so the operation reported 2 diagnostic model errors and skipped no paragraphs;
- Character Bible page loaded the generated bible and showed 23 characters.

After that UI run, the rare `identity_conflict` split branch was connected so it calls `SplitCandidateAgent` and records the split proposal without applying it automatically. Build and non-manual tests passed after that final code change; a third UI run was not started because the local command approval system rejected starting another Web process due to the current usage limit.

Final contract reconciliation added `aliasesToAdd[]` to `DossierPatchProposal`, kept the prompt in a Markdown resource, and applies approved alias additions only from candidate alias evidence. Verification passed:
