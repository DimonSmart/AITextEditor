# 0008 SPEC Book file and bible loading

Type: spec

Replaces:
- none

Related:
- 0007

## Goal

Add explicit UI actions for opening and saving a Markdown book file, and keep the book's character bible file loaded from and saved to a predictable companion Markdown path.

## Context

The Blazor editor UI currently focuses on a single editor session and character bible projection. Users need to load an arbitrary book from disk, save edits back to disk, and have the matching character bible appear immediately with that book. The editor layout should use the native MudBlazor splitter component instead of a custom splitter implementation.

## Scope

- Add a Book UI action to open and load a Markdown book file.
- Add a Book UI action to save the current book text.
- Use a companion character bible path derived from the book path.
- Load the companion character bible automatically when a book is opened.
- Save the companion character bible with the current book using the same derived path.
- Replace the current custom splitter with the real native MudBlazor splitter component.

## Non-goals

- Do not add multi-book workspace support.
- Do not add compatibility paths for older companion bible names.
- Do not add Git/versioning UI.
- Do not introduce a second character bible data model.
- Do not parse arbitrary character bible Markdown into structured dossiers unless the current code already has a direct supported path.

## Plan

1. Inspect the current Blazor editor state, Book page, character bible services, and MudBlazor packages.
2. Add file open/save plumbing using the existing app patterns.
3. Add deterministic companion bible path handling, using `<book-name>-character-bible.md` because the codebase and UI use the term `Character Bible`.
4. Replace the splitter markup with the native MudBlazor splitter.
5. Run repository checks and a local Web build/start smoke check when practical.

## Pause points

- Stop if MudBlazor in the project version does not provide a native splitter component and adding or upgrading the package would require broad unrelated changes.
- Stop if the current character bible layer cannot load or persist Markdown without inventing a new persistence model.

## Acceptance criteria

- The Book page has clear open/load and save actions for Markdown book files.
- Opening a book updates the editor and automatically attempts to load the derived character bible file.
- Saving writes the book and the companion bible file to the derived path.
- The derived bible file name is predictable from the book file name.
- The editor split layout uses the native MudBlazor splitter component.
- Existing non-manual tests still pass.

## Verification

Commands or manual checks:

```bash
dotnet build AiTextEditor.sln
dotnet test --filter "Category!=Manual"
dotnet run --project src/AiTextEditor.Web
```

## Outcome

The companion file uses `<book-name>-character-bible.md` instead of `<book-name>-bible-heroes.md`. The codebase and UI consistently use `Character Bible`, and `bible-heroes` is ambiguous in English. The file source of truth is a fenced YAML dossiers block inside the Markdown file, followed by the rendered Markdown projection. This keeps the file readable as Markdown while avoiding unsupported parsing of arbitrary prose Markdown back into structured dossiers.

The native splitter required `MudBlazor` 9.0.0; `MudBlazor` 8.x in the local cache did not include `MudSplitPanel`. The Web project now references MudBlazor 9.0.0 and uses `MudSplitPanel` in the shared Markdown editor.

Verification completed:

```bash
dotnet build AiTextEditor.sln
dotnet test --filter "Category!=Manual"
```

Manual smoke check returned HTTP 200 for `http://localhost:5077/book`. Browser interaction loaded `C:\Private\AITextEditor\README.md`, enabled Save, updated the document id to `README`, and showed the derived companion path `C:\Private\AITextEditor\README-character-bible.md`. Browser inspection also showed the `Resize panels` separator from `MudSplitPanel`.
