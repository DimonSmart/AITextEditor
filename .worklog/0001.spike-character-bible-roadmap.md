# 0001 SPIKE Character bible roadmap

Type: spike

Replaces:
- none

Related:
- none

## Goal

Understand the current project state and propose a practical path toward an automatic character bible pipeline for long Markdown manuscripts.

## Hypothesis

The project can reach the described target state through a staged workflow: deterministic Markdown traversal first, narrow interpretation agents for extraction and matching, a single canonical Markdown write path, and a later consolidation pass for ambiguous entities.

## Experiment

Check the current repository structure, existing app capabilities, available MCP or Markdown editing pieces, and any planning notes that already describe multi-agent processing.

Compare the current state with the proposed architecture and identify:

- what already exists;
- what is missing;
- what should be built first;
- which parts are research spikes versus implementation specs;
- which verification commands should guard the next implementation step.

## Constraints

- Do not implement the architecture during this spike.
- Keep recommendations scoped to this repository and its current technology stack.
- Prefer deterministic code for traversal, persistence, and commits.
- Use agents only where interpretation is needed.
- Keep one clear write path for canonical Markdown updates.

## Done when

- current state is summarized;
- architectural gaps are documented;
- staged implementation plan is proposed;
- follow-up work documents are clear if needed;
- recommendation is written.

## Result

Checked the repository state on 2026-05-21.

Existing pieces:

- Markdown is parsed into a linear document with semantic pointers.
- The editor session exposes deterministic load, inspect, target set, and linear edit operations.
- Streaming cursor infrastructure already supports full-scan, keyword, and cursor-agent navigation over long Markdown documents.
- Character dossier generation already exists as an in-memory Semantic Kernel plugin. It scans document portions, calls one LLM extractor, then applies deterministic name/alias matching into `CharacterDossierService`.
- Dossier storage can serialize JSON/YAML internally, but the public workflow still treats dossiers as in-memory state rather than a canonical Markdown artifact.

Main gaps against the target architecture:

- There is no Microsoft Agent Framework workflow or other explicit graph-level orchestration yet; current orchestration is Semantic Kernel auto tool invocation plus hand-written loops.
- Book traversal and character dossier generation are still coupled inside `CharacterDossiersGenerator`.
- Candidate extraction, matching, resolving, committing, and consolidation are not separate contracts.
- The current model has no status field for `canonical`, `provisional`, `merged`, or `rejected`, and no persisted ambiguity bucket.
- Matching is mostly name/alias-key based. It does not produce top-k candidates, confidence, or explicit `existing/new/ambiguous/defer` decisions.
- The write path for character dossiers is not yet a single canonical Markdown committer.
- Checkpointing, resumability, durable session state, and post-pass consolidation are not implemented.
- Telemetry exists as logging, but there is no first-class trace of extraction/matching/commit decisions.

## Recommendation

Do not start by replacing the whole project with a large multi-agent system. The repository already has the useful lower layers: Markdown linearization, semantic pointers, cursor streams, and an initial dossier generator. The next step should be a narrow `CharacterBibleWorkflow` capability that keeps traversal and writes deterministic while splitting LLM work into small typed decisions.

Recommended sequence:

1. Add explicit domain contracts for character mentions, candidate matches, resolver decisions, dossier status, ambiguity records, and commit proposals.
2. Extract the current traversal loop from `CharacterDossiersGenerator` into a deterministic chunk cursor service that emits stable Markdown chunks with pointer ranges and optional carry-forward context.
3. Convert the existing prompt into a `CandidateExtractor` that returns mention candidates, not final dossiers.
4. Add a deterministic matcher that builds top-k existing dossier candidates from names, aliases, pointer proximity, recent active characters, and later embeddings.
5. Add a resolver step that returns only `existing`, `new`, `ambiguous`, or `defer` decisions with reasons and evidence pointers.
6. Add a single committer that owns all changes to the canonical character bible. Initially it can write through `CharacterDossierService`; then it should persist a Markdown source of truth plus a machine sidecar index.
7. Add post-pass consolidation over unresolved and near-duplicate records after the full manuscript pass.
8. Only after those contracts are stable, decide whether to migrate orchestration from Semantic Kernel to Microsoft Agent Framework workflows. The migration should be isolated behind the same domain contracts.

## Follow-up

- Create a spec for the first `CharacterBibleWorkflow` MVP.
- Create an ADR for canonical Markdown plus sidecar index storage.
- Create a spike for Microsoft Agent Framework migration only after the MVP contracts are clear.
